# Functional Specification

## 1. 문서 목적
본 문서는 원격 데스크톱 연결 프로그램의 주요 기능을 구현 단위로 세분화하고, 각 기능의 입력, 처리, 출력, 예외 상황을 정의하기 위한 기능 명세서이다.

## 2. 시스템 구성 개요
- Client App: 원격 장치에 접속하는 사용자 애플리케이션(.NET 9, x64 전용)
- Host Agent: 제어 대상 PC에서 실행되는 에이전트 또는 서비스
- Session Manager: 연결 상태, 인증, 세션 수명주기를 관리하는 모듈
- Transport Layer: 화면 데이터, 입력 이벤트, 파일 전송 데이터를 전달하는 통신 계층
- Audit Logger: 접속 및 주요 동작 로그를 기록하는 모듈

## 2.1 클라이언트 아키텍처 원칙
- 클라이언트 애플리케이션은 .NET 9 기반으로 구현한다.
- 초기 배포 대상은 x64 아키텍처로 한정한다.
- UI 계층은 MVVM 패턴을 적용한다.
- View는 화면 렌더링과 사용자 입력 바인딩에 집중한다.
- ViewModel은 화면 상태, 명령, 세션 흐름 제어를 담당한다.
- Model 및 Service 계층은 장치, 세션, 인증, 전송, 로그 등 도메인 로직을 담당한다.
- 코드 비하인드에는 화면 표시와 직접 관련된 최소 로직만 허용한다.

## 3. 기능 상세
### 3.1 장치 등록
- 목적: 사용자가 연결 가능한 원격 장치를 식별하고 목록화할 수 있도록 한다.
- UI 책임:
  - View는 장치 목록과 등록 폼을 표시한다.
  - ViewModel은 등록 명령, 입력 검증, 결과 상태 갱신을 담당한다.
- 입력:
  - 장치 이름
  - 장치 ID 또는 등록 코드
  - 장치 설명(optional)
- 처리:
  - 시스템은 장치 식별자를 검증한다.
  - 유효한 장치인 경우 사용자 계정 또는 로컬 저장소에 등록한다.
- 출력:
  - 등록 성공 메시지
  - 장치 목록 반영
- 예외:
  - 중복 장치 등록
  - 유효하지 않은 장치 코드
  - 등록 중 네트워크 실패

### 3.2 연결 요청 및 승인
- 목적: 사용자가 원격 장치에 접속을 요청하고, 대상 장치에서 승인을 받아 세션을 생성한다.
- UI 책임:
  - View는 연결 진행 상태와 승인 대기 화면을 표시한다.
  - ViewModel은 연결 요청, 취소, 재시도, 상태 전이를 관리한다.
- 입력:
  - 대상 장치 ID
  - 인증 수단(PIN, 세션 코드 등)
- 처리:
  - 시스템은 장치 연결 가능 상태를 확인한다.
  - Host Agent는 사용자에게 접속 요청 팝업 또는 승인 UI를 노출한다.
  - 승인 성공 시 세션을 생성한다.
- 출력:
  - 연결 중 상태
  - 승인 대기 상태
  - 연결 성공 또는 실패 결과
- 예외:
  - 대상 장치 오프라인
  - 승인 거부
  - 인증 실패
  - 시간 초과

### 3.3 원격 화면 전송
- 목적: 사용자가 원격 PC 화면을 실시간으로 확인할 수 있도록 한다.
- UI 책임:
  - View는 원격 화면 및 상태 오버레이를 렌더링한다.
  - ViewModel은 화면 품질 옵션과 세션 상태 바인딩을 관리한다.
- 입력:
  - 세션 ID
  - 화질 또는 해상도 옵션
- 처리:
  - Host Agent는 화면을 캡처하여 인코딩 후 전송한다.
  - Client App은 수신 데이터를 디코딩하고 렌더링한다.
- 출력:
  - 원격 화면 표시
  - 품질 상태 또는 지연 상태 표시
- 예외:
  - 캡처 실패
  - 프레임 전송 지연
  - 화면 렌더링 오류

### 3.4 입력 제어
- 목적: 사용자가 마우스와 키보드로 원격 장치를 제어할 수 있도록 한다.
- UI 책임:
  - View는 로컬 입력 이벤트를 수집한다.
  - ViewModel은 입력 활성화 상태와 세션 제어 가능 여부를 관리한다.
- 입력:
  - 마우스 이동/클릭 이벤트
  - 키보드 입력 이벤트
- 처리:
  - Client App은 입력 이벤트를 전송한다.
  - Host Agent는 입력을 OS 수준에서 반영한다.
- 출력:
  - 화면 반영 결과
- 예외:
  - 입력 권한 없음
  - 시스템 단축키 제약
  - 관리자 권한 부족

### 3.5 파일 전송
- 목적: 로컬 장치와 원격 장치 간 파일 교환을 지원한다.
- UI 책임:
  - View는 파일 선택, 진행률, 결과 메시지를 표시한다.
  - ViewModel은 전송 시작, 취소, 진행 상태 갱신을 담당한다.
- 입력:
  - 파일 경로
  - 전송 방향(업로드/다운로드)
- 처리:
  - 파일 크기와 전송 가능 여부를 확인한다.
  - 세션 채널 또는 별도 데이터 채널로 전송한다.
  - 다운로드 시 원격 파일 메타데이터(파일명, 크기)를 먼저 수신하고, 로컬 저장 경로는 안전한 파일명 규칙과 중복 회피 규칙에 따라 결정한다.
- 출력:
  - 전송 진행률
  - 성공/실패 결과
- 예외:
  - 권한 부족
  - 파일 크기 제한 초과
  - 네트워크 단절

### 3.6 클립보드 공유
- 목적: 세션 중 텍스트 기반 클립보드를 양방향으로 공유한다.
- UI 책임:
  - ViewModel은 클립보드 동기화 활성화 상태를 관리한다.
- 입력:
  - 로컬 또는 원격의 텍스트 클립보드 데이터
- 처리:
  - 클립보드 변경 이벤트를 감지하고 동기화한다.
  - 동일 텍스트 재전송과 재적용을 방지하기 위해 마지막 송신 텍스트와 마지막 수신 적용 텍스트를 함께 비교한다.
  - MVP 단계에서는 텍스트 경로를 우선 보장하고, 비정상적으로 큰 텍스트 payload는 방어적으로 무시한다.
- 출력:
  - 복사/붙여넣기 가능 상태
- 예외:
  - 형식 미지원
  - 보안 정책으로 인한 제한

### 3.7 자동 재연결
- 목적: 네트워크 장애 시 세션 복구를 시도한다.
- UI 책임:
  - View는 재연결 진행 상태와 실패 메시지를 표시한다.
  - ViewModel은 재시도 횟수, 백오프 상태, 사용자 후속 액션을 관리한다.
- 입력:
  - 기존 세션 정보
  - 재연결 시도 횟수 및 정책
- 처리:
  - 연결 끊김 감지 후 일정 시간 동안 재시도한다.
  - 성공 시 세션을 복구한다.
- 출력:
  - 재연결 시도 상태
  - 복구 성공 또는 실패 안내
- 예외:
  - 장시간 오프라인
  - 세션 만료

### 3.8 로그 및 감사
- 목적: 접속 이력과 주요 동작을 기록해 추적성을 확보한다.
- UI 책임:
  - View는 세션 이력 목록과 상세 상태를 표시한다.
  - ViewModel은 필터링, 정렬, 조회 요청을 관리한다.
- 입력:
  - 접속 시각
  - 사용자
  - 장치
  - 승인 결과
  - 종료 사유
- 처리:
  - 로컬 또는 서버 저장소에 로그를 기록한다.
  - 민감 데이터는 최소화 또는 마스킹 처리한다.
  - 로컬 저장소 로그 파일이 손상되었을 경우 앱 기동은 계속되어야 하며, 손상 파일은 백업 후 새 로그 저장소로 복구한다.
- 출력:
  - 세션 이력 조회 정보
- 예외:
  - 로그 저장 실패
  - 저장소 용량 부족

## 4. 공통 예외 처리 원칙
- 사용자가 이해할 수 있는 오류 메시지를 제공해야 한다.
- 네트워크 장애, 인증 실패, 권한 부족, 버전 불일치 상황을 구분해서 안내해야 한다.
- 오류 발생 시 가능한 복구 행동을 함께 제시해야 한다.

## 5. 보안 요구사항
- 세션 연결 전 인증이 완료되어야 한다.
- 승인 기반 접속에서는 Host 측의 명시적 동의가 필요하다.
- 전송 데이터는 암호화되어야 한다.
- 비밀번호, PIN, 토큰 등 민감 정보는 평문으로 저장하지 않는다.
- 로그에는 민감 입력값을 남기지 않는다.
- 자가 서명 인증서를 사용하는 경우에도 인증서 지문은 재시작 후 유지되어야 하며, 저장된 지문과 실제 지문이 다르면 연결을 차단해야 한다.

## 6. 향후 기술 설계 시 확인 필요 항목
- P2P와 Relay 혼합 구조 여부
- Host Agent 실행 방식(백그라운드 서비스/트레이 앱)
- 화면 캡처 기술 선택
- 입력 주입 방식
- 파일 전송 채널 분리 여부
- 계정 기반 인증 확장 방식
- MVVM 기준 화면 모듈 분리 범위 및 공통 ViewModel 베이스 구조
