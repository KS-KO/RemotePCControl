# Development Schedule

## 1. 문서 목적
본 문서는 원격 데스크톱 연결 프로그램의 초기 개발 일정을 단계별로 정리한 계획 문서이다. 일정은 MVP 기준이며, 실제 인력 구성과 기술 검토 결과에 따라 조정될 수 있다.

## 2. 전체 일정 가정
- 기준 버전: MVP 1차 출시
- 예상 기간: 12주
- 전제 조건:
  - Windows 우선 개발
  - .NET 9 기반 개발
  - x64 전용 빌드
  - 클라이언트 UI는 MVVM 패턴 적용
  - 소규모 팀 기준
  - 핵심 연결 기능 우선

## 3. 마일스톤별 일정
### 1주차-2주차: 요구사항 확정 및 기술 검토
- PRD 확정
- MVP 범위 확정
- 연결 구조 검토(P2P, Relay, Hybrid)
- 보안 요구사항 정리
- 화면 캡처 및 입력 제어 방식 조사
- .NET 9 및 x64 배포 기준 확정
- MVVM 기반 클라이언트 구조 설계
- 기본 아키텍처 초안 작성

### 3주차-4주차: 기반 구조 설계 및 개발 환경 구축
- 프로젝트 구조 설계
- Host/Client 기본 모듈 분리
- 세션 관리 구조 정의
- 로그 구조 정의
- View, ViewModel, Model/Service 레이어 분리 기준 정의
- 공통 ViewModel 베이스 및 커맨드 구조 설계
- 개발/빌드 환경 세팅
- x64 타깃 빌드 및 배포 설정
- 기본 UI 와이어프레임 정리

### 5주차-7주차: 핵심 연결 기능 개발
- 장치 등록 기능 구현
- 연결 요청/승인 플로우 구현
- 세션 생성 및 종료 처리 구현
- 화면 캡처 및 전송 프로토타입 구현
- 입력 제어 구현
- MVVM 바인딩 기반 메인 화면 및 세션 화면 구현

### 8주차-9주차: 부가 기능 및 안정화

## 3.1 현재 기준 잔여 개발 단계
### Phase A: 검증 및 문서 고정
- `RESILIENCE_REVIEW.md` 체크리스트 기준 실제 회복탄력성 테스트 수행
- 보안 시나리오 기준 최초 신뢰 등록, 지문 불일치 차단, 재시작 후 인증서 유지 검증
- 테스트 결과를 PRD 및 Resilience 문서의 기록 표에 반영

### Phase B: 연결 승인 워크플로우 고도화 (Completed)
- `Support request`를 host-driven 승인/응답 채널로 확장 (Done)
- 승인 결과와 보안 신뢰 흐름을 하나의 운영 시나리오로 정리 (Done)
- 세션 상태 텍스트와 감사 로그의 운영 문구 정제 (Done)

### Phase C: 사용자 경험 고도화 (Progressing)
- measured latency 및 품질 상태 표시 추가 (Done)
- 파일 전송 UX 개선: Ctrl+C/V 기반 자동 업로드/다운로드 핸들링(FR-8) 구현 완료 (Done)
- 로컬 드라이브 리디렉션의 폴더 선택/노출 범위 제어 고도화

### Phase D: 확장 구조 결정
- 연결 구조를 `P2P`, `Relay`, `Hybrid` 중 하나로 확정
- 인터넷 환경 확장을 위한 디렉터리/중계 구조 설계
- 무인 접속과 계정 기반 인증의 MVP 이후 분리 범위 확정

## 6. 일정 리스크
- 화면 캡처 및 입력 제어 기술 검증이 예상보다 오래 걸릴 수 있다.
- 네트워크 구조 결정이 늦어지면 전체 구현 일정이 밀릴 수 있다.
- 파일 전송 및 재연결 안정화에 추가 시간이 필요할 수 있다.
- 보안 요구사항 강화 시 인증 및 저장 구조 수정이 필요할 수 있다.
- MVVM 구조 분리가 미흡하면 초기 속도는 빨라도 이후 유지보수 비용이 증가할 수 있다.

## 7. 권장 운영 방식
- 주간 단위로 기능 데모를 진행한다.
- 2주차 종료 시 MVP 범위를 동결한다.
- 7주차 종료 시 핵심 연결 흐름의 동작 여부를 반드시 검증한다.
- 10주차부터는 신규 기능보다 안정화와 테스트를 우선한다.
- 현재 시점부터는 신규 기능 추가보다 `검증 -> 운영 문구 정리 -> 남은 host workflow 완성` 순서로 진행한다.
